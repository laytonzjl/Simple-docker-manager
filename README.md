# Simple-docker-manager
一个简单的服务器docker创建、配置和管理脚本，玩具级别，仅限能用。

# 实验室 GPU 容器管理系统部署指南

本教程将带你一步步将容器管理脚本部署到服务器中，并将其配置为全局命令，实现开箱即用的自动化运维。

---

## 🛠️ 详细部署步骤

### 步骤 1：进入专用管理目录
首先，切换到宿主机的专用管理目录 `/opt/container_manager`。
```bash
cd /opt/container_manager
```
* **💡 为什么这么做？**
    在 Linux 系统中，`/opt` 是专门用来存放第三方独立软件和自定义脚本的规范位置。把脚本放在这里，既能保证其全局安全性，又不会与操作系统的原生文件混淆，避免被误删。

---

### 步骤 2：创建并打开脚本文件
使用系统自带的轻量级文本编辑器 `nano` 创建并打开一个名为 `lab_manager.sh` 的空白文件：
```bash
sudo nano lab_manager.sh
```
* **💡 为什么这么做？**
    长篇幅的 Bash 脚本（包含大量的 `$` 符和结构化缩进）如果直接用 `vim` 粘贴，经常会被 Vim 误判为快捷键或触发混乱的自动缩进，导致代码严重错乱。而 `nano` 是一个纯净的“所见即所得”编辑器，鼠标右键粘贴时绝对不会对代码进行二次篡改。

---

### 步骤 3：粘贴代码、保存并退出
1. **复制脚本**：复制你下载的docker-manager.sh里面的完整代码。
2. **粘贴代码**：在打开的黑色 `nano` 界面中，点击**鼠标右键**完成粘贴（或使用快捷键 `Ctrl + V`）。
3. **触发退出**：按下键盘上的 **`Ctrl + X`**（代表退出 Exit）。
4. **确认保存**：此时底部状态栏会询问 *Save modified buffer? (是否保存修改？)*，请在键盘上敲击字母 **`Y`**（代表 Yes）。
---

### 步骤 4：赋予脚本可执行权限
新创建的文件默认只有读写权限，无法作为程序被直接运行：
```bash
sudo chmod +x /opt/container_manager/lab_manager.sh
```
* **💡 为什么这么做？**
    如果不跑这一步，Linux 操作系统会出于安全机制将该脚本锁定。后续即使你做了快捷方式，系统在调用时也会由于权限不足而当场抛出 `Permission denied` 或 `command not found` 的致命报错。

---

### 步骤 5：创建全局系统软链接
为了不让管理员每次都敲长长一串路径，我们需要将脚本“挂载”到操作系统的全局二进制搜索路径中：
```bash
sudo ln -s /opt/container_manager/lab_manager.sh /usr/local/bin/docker-manager
```
* **💡 为什么这么做？**
    `/usr/local/bin` 是 Linux 环境变量 `$PATH` 里的核心常驻路径。只要把快捷方式（软链接）扔进这个文件夹，系统就会把我们的脚本当成一个和 `cd`、`ls`、`docker` 一样级别的“系统内置命令”。

---

### 步骤 6：在全系统任意位置启动
大功告成！现在，无论你在服务器的哪一个文件夹下（比如根目录 `/`、或者是普通用户的Home目录 `~/`），只需要直接在终端敲下这行命令，就能瞬间唤醒实验室管理系统：
```bash
sudo docker-manager
```

## 🛑 常见故障排查 (Troubleshooting)

### Q1: 提示 `ln: 无法创建符号链接 '/usr/local/bin/docker-manager': 文件已存在`
* **原因**：说明你之前已经创建过同名的快捷方式。
* **解决**：先运行 `sudo rm -f /usr/local/bin/docker-manager` 删掉老链接，再重新跑 **步骤 5** 即可。

### Q2: 提示 `sudo: docker-manager: command not found`
* **原因**：大概率是因为你漏掉了 **步骤 4**（未加可执行权限），或者你把快捷方式指向了一个错误或打错字的文件路径（形成了死链接）。
* **解决**：重新执行一遍 **步骤 4** 确保权限给齐；如果还不行，执行 `ls -l /usr/local/bin/docker-manager` 查看箭头的指向是否与你的真实文件路径完全一致。
